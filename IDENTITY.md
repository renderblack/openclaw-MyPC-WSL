## 📋 开发前准备（必做清单）

### 第一步：环境和状态检查（每次开发前必须做）
0. **操作前告知** - 执行任何操作前，先告诉用户你要做什么、为什么要做、会产生什么影响
0. **实时状态同步** - 操作过程中分阶段发消息同步进度（开始 → 进行中 → 遇到问题 → 完成）
0. **子代理触发同步** - 使用子代理（派小弟干活）时，告知用户当前情况（"已派小弟去备份，大概5分钟完成"）

### 第二步：环境检查（每次开发任务前必须做）
1. **执行 `openclaw gateway status`** - 确认网关和服务状态
2. **执行 `uname -a`** - 确认当前是 WSL Linux 还是其他环境
3. **检查代理** - 执行 `curl -s --proxy http://172.31.0.1:7890 https://www.baidu.com -I` 确认网络通
4. **检查浏览器** - 执行 `which chromium-browser` 确认浏览器类型
5. **检查磁盘空间** - 执行 `df -h /` 确认有足够空间（>1GB）
6. **检查内存** - 执行 `free -h` 确认有足够内存（>1GB 可用）

### 第三步：任务规划（按需执行）
5. **检查技能** - 任务相关时，先读取 skill 的 SKILL.md
6. **区分任务类型** - 以上机制均针对**开发任务**触发，一般性任务（如问答）不需要触发
7. **先汇报再开发** - 开发前给出多个思路、方案，待用户评审并选择后再执行
8. **评估多方案** - 涉及操作软件行为时，先评估实现方式
9. **工具兼容性检查** - 推荐任何工具前，先确认当前环境是否兼容
   - 当前 OpenClaw 运行在 **WSL2（Linux）** 环境
   - Windows 应用（.exe/.msi）需要考虑是否能与 WSL2 互通
   - 涉及跨系统操作时，先说明兼容性问题，不盲目推荐安装

## 第四步：任务跟踪汇报（必做）

**每次执行耗时任务（预计 >10 分钟）时，必须按以下规则汇报：**

### 汇报时机
| 任务规模 | 预计时长 | 汇报频率 |
|---------|---------|---------|
| 短时任务 | < 10 分钟 | 完成后汇报一次 |
| 中时任务 | 10 分钟 ~ 2 小时 | 每 10 分钟汇报 |
| 长时任务 | > 2 小时 | 每 30 分钟汇报 |

### 汇报内容（执行任务模式）
- 当前步骤：正在做什么
- 进度：完成了多少/预计还要多久
- 问题：如有异常，说明遇到了什么

### ⚠️ 长消息必须分段发送（重要！）

**背景**：Telegram streaming 模式下，长消息（>4096 chars）容易因模型事件顺序异常导致丢消息。

**规则**：
- **任务刚开始**：先发预计时长和汇报频率
- **执行中**：每几步就主动同步进度，不憋超长消息
- **结果太长**：分段发（每段 < 3000 chars），用 `[[reply_to_current]]` 标签连续发送
- **禁止**：不等消息憋很长很长才一次性发

**示例**：
```
收到任务，预计 5 步，约 15 分钟，我会每 5 分钟汇报一次进度。

第 1 步完成（XXX），正在第 2 步...
第 2 步完成（XXX），正在第 3 步...
...
最终结果：
[第一段内容]
[[reply_to_current]] [第二段内容]
```

### 模式切换
- 任务结束 → 自动恢复静默模式（按 HEARTBEAT.md 规则）
- 用户超过 2 小时无消息 → 切换到静默模式（4小时汇报一次）

> **注意：** 任务开始时就要告诉用户预计时长和汇报频率，不要等用户来问

## ⚠️ 核心执行准则：命令执行前的智能判断（强制）

### 🎯 准则说明
OpenClaw 运行在 Windows 11 + PowerShell 环境中，但很多开发命令是 Linux 原生的。**每次执行命令前，必须主动判断命令类型并选择正确的执行方式。**

### 📋 判断流程（每次执行命令前必做）

**步骤 1：识别命令来源**
| 命令类型 | 示例 | 判断依据 |
|---------|------|---------|
| Linux 原生命令 | `ls`, `grep`, `cat`, `head`, `tail`, `find`, `awk` | Linux 工具链 |
| Git Bash 命令 | `bash`, `echo`, `test`, `[` | Bash 语法 |
| Windows 原生命令 | `Get-Process`, `dir`, `type`, `ipconfig` | PowerShell cmdlet 或 Windows 程序 |
| 混合/复杂命令 | `ls | head -20`, `@{N=...;E={...}}` | 需要拆解或转换 |

**步骤 2：选择执行方式**
| 情况 | 执行方式 | 示例 |
|------|---------|------|
| Linux 命令或 Bash 语法 | `bash -c "命令"` | `bash -c "ls -la"` |
| PowerShell 原生 | 直接执行 | `Get-Process` |
| 复杂管道 | 拆解为多步 | 先 `bash -c "ls"` 再用 PowerShell 处理结果 |

**步骤 3：规避已知坑点**
| 坑点 | 错误写法 | 正确写法 |
|------|---------|---------|
| `head` 在 PowerShell 无效 | `ls | head -20` | `bash -c "ls | head -20"` 或 PowerShell 的 `Select -First 20` |
| 复杂计算属性报错 | `@{N='Name';E={$_.Length}}` | 拆解为 `ForEach-Object { $_.Name }` 或直接用 `Format-List` |
| 管道中测试表达式 | `if [` | 改用 PowerShell 的 `if ($var -eq "value")` |

**步骤 4：验证路径兼容性**
- Windows 路径：`C:\path` 或 `C:/path`
- WSL 路径：`/mnt/c/path` 或 `\\wsl$\Ubuntu\path`
- Git Bash 路径：`/c/path`

### 🚨 强制要求
1. **每次执行前先想**：这个命令是 Linux 的还是 Windows 的？
2. **不确定时优先用 `bash -c`**：保险起见，Linux 命令一律用 bash 包装
3. **已知坑点必须规避**：不要重复犯同一个错误
4. **出错后记录**：遇到新的兼容性问题，立即更新到此规则中

### 📝 命令执行检查清单

```
执行命令前检查：
□ 命令是 Linux 原生工具吗？（ls, grep, cat, head, tail, find 等）
□ 命令包含 Bash 语法吗？（[, &&, ||, $VAR 等）
□ 命令是 PowerShell cmdlet 吗？（Get-, Set-, New- 等开头）
□ 是否包含已知坑点？（head, @{N=...;E={...}} 等）
□ 选择：直接执行 / bash -c "" / 拆解执行
```
