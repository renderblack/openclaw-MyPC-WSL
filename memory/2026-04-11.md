## 下午讨论：WSL2能看到Windows界面方案分析

### 目标
WSL2里的OpenClaw能"看到"并"操作"Windows浏览器/桌面

### 发现的关键事实
1. agent-browser启动的是WSL2内部的Linux Chrome（headless），不是Windows Chrome
2. Windows Chrome调试端口(9222/9223)从WSL2可达，但agent-browser --cdp只连本地127.0.0.1
3. VcXsrv是WSL2→Windows单向显示，无法捕获Windows桌面内容
4. 两个Chrome互相独立，无法通过VcXsrv桥接

### 方案对比
| 方案 | 原理 | 难度 | 可行性 |
|------|------|------|--------|
| VcXsrv X11 | WSL2程序显示到Windows | 低 | ❌ 只是单向显示WSL2程序 |
| Extension Relay | 浏览器扩展推送内容 | 高 | 待探索 |
| CDP端口转发 | 连接Windows Chrome CDP | 中 | 需socat端口转发 |
| VNC | 捕获整个Windows桌面 | 中 | ✅ 直接可行 |

### 决定的下一步
1. **方案3 VNC** → 先尝试（在Windows装VNC Server，WSL2用VNC Client连接）
2. **方案1 Extension Relay** → 之后尝试

### VNC准备检查
- Windows是否已有VNC（RealVNC/TightVNC/AnyDesk/ToDesk）：待确认
- WSL2是否有VNC Client：待确认

## 下午探索：WSL2能看到Windows界面方案（未完成）

### 目标
让WSL2里的OpenClaw能"看到"并"操作"Windows浏览器界面

### 已尝试的方案

#### 1. agent-browser（WSL2 Chrome）
- 状态：✅ 可用，能操作+截图
- 问题：WSL2里的headless Chrome老大看不到窗口
- 结论：只能通过截图传递画面

#### 2. VcXsrv X11转发
- 状态：VcXsrv安装成功，X11端口(6000)连接成功
- 问题：WSL2里Xvfb崩溃（NVIDIA WSL驱动冲突）
- 结论：方案卡住

#### 3. VNC远程桌面
- 状态：TightVNC在Windows上安装成功，端口5900可达
- 问题：WSL2里没有VNC Client（需要安装tigervnc-viewer）
- further问题：Xvfb崩溃导致无法渲染VNC画面
- 结论：方案卡住

#### 4. Browser Relay扩展
- 状态：发现官方文档，但`openclaw browser extension install`命令不存在
- 问题：
  1. Chrome扩展文件在npm包里找不到
  2. 端口是18792不是18793
  3. `extension install`子命令版本不支持
- 你Chrome上装的"OpenClaw Browser Relay"来源不明

#### 5. openclaw-browser-channel npm包
- 状态：安装失败（依赖太多）
- 结论：第三方包，不可用

### 关键发现

1. **agent-browser启动的是WSL2内部的Linux Chrome**，不是Windows Chrome
2. **Windows Chrome调试端口(9222/9223)可达**，但agent-browser --cdp只连本地
3. **VcXsrv/Xvfb崩溃根因**：WSL2的NVIDIA驱动和Xvfb不兼容
4. **Browser Relay官方存在**：https://openclawlab.com/en/docs/tools/chrome-extension/
   - 中继端口：18792（不是18793）
   - 需要chrome profile连接中继
