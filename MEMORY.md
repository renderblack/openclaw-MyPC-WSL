## 📝 工作原则

### 1. 分步骤执行 + 实时同步 (2026-03-28 确立)
- **要求**: 每一步都详细阐述逻辑操作流程，不闷头干完才给结果
- **原因**: 用户想通过过程学习，参与技术决策，避免在黑暗中等待
- **示例流程**:
  1. "收到，开始做，预计3步：1、2、3"
  2. "开始第1步：XXX，因为..."
  3. "第1步完成（结果/发现），正在第2步..."
  4. "完成了，结果是XXX"

### 2. 追求自主操作能力 (2026-03-28 确立)
- **核心目的**: 实现 AI 能够自主操作电脑，而不是让用户手动操作
- **态度**: 遇到限制就找替代方案，一步一步实现，不追求一步到位
- **例子**: WSL 调用 Windows GUI 的三步走策略

### 3. 命令执行前的智能判断 (2026-04-04 确立)
- **问题**: OpenClaw 运行在 Windows PowerShell，但很多命令是 Linux 原生
- **判断流程**:
  1. 识别命令来源 (Linux/Bash vs PowerShell)
  2. 选择执行方式 (`bash -c` 包装 Linux 命令)
  3. 规避已知坑点 (`head`, `@{N=...;E={...}}` 等)
- **强制要求**: 每次执行前先想"这个命令是 Linux 的还是 Windows 的？"

### 4. 环境兼容性检查 (2026-03-28 确立)
- **原则**: 推荐/安装任何工具前，先检查环境兼容性
- **常见问题**: WSL2 vs Windows 应用兼容性
- **教训**: 曾推荐 CC-Switch (Windows 应用) 给 WSL2 环境，导致无法连接

### 消息格式要求（强制）
**每次回复必须遵守以下格式，避免“新建对话就忘记”的问题：**

1. **分块清晰**：
   - 使用 `##` 标题区分大模块
   - 使用 `-` 列表展示步骤/要点
   - 使用代码块展示命令/日志

2. **重点突出**：
   - 关键数据用 **粗体**
   - 警告/建议用列表或单独段落
   - 避免大段纯文本堆砌

3. **少用 Emoji**：
   - 仅在标题或关键结论处使用（如 ✅, ⚠️, 🔴）
   - 避免在普通文本中大量使用（防止乱码）

4. **逻辑连贯**：
   - 先说结论（结论先行）
   - 再解释原因（Why）
   - 最后给建议（How）

5. **禁止事项**：
   - ❌ 禁止 Markdown 表格（QQ 端易乱码，改用列表）
   - ❌ 禁止超长段落（超过 3 行需换行）
   - ❌ 禁止“乱码式”输出（如乱码的中文/Emoji）

### 执行原则
- **每次会话启动时**：读取 `MEMORY.md` 确认格式规范
- **每次回复前**：自检是否符合上述 5 点
